# External CFD / PIV / 3D validation specification

Status: **research specification** for a post-V29 external-validation and 3D
research layer. This document does not define a certified scientific revision and
does not modify `ITD V29.18`. The certified baseline (`MODEL_REVISION`,
equations, expected outputs, analytical oracles, reference summaries, Rust
fixture) remains unchanged. All new work lives outside `itd_v29_core`, under the
isolated `itd_research/` namespace.

## 0. Terminology and honesty rules

The work in this phase is labelled an **external-validation research candidate**
and an **experimental 3D ITD candidate**. It is not `V29.19`, `V30`, or any
certified revision. The following distinctions are kept explicit throughout:

* **code verification** — the implementation computes the intended formula
  (checked against exact analytical inputs);
* **numerical validation** — discrete results converge to analytical references;
* **comparison with independent CFD** — results on data generated by a solver
  that is not ITD;
* **comparison with experimental PIV** — results on measured velocity fields;
* **qualitative vs quantitative agreement**;
* **statistical association vs causal interpretation**.

Synthetic and manufactured fixtures are used only for code verification,
numerical validation, and CI. **A synthetic fixture is never presented as
external empirical validation.** Empirical claims require genuinely independent
CFD or experimental data with complete provenance.

### Environment constraints (recorded honestly)

The development environment for this repository has **no OpenFOAM binaries** and
**no VTK/HDF5/NetCDF libraries** installed, and CI must not depend on third-party
servers. Consequently:

* CFD cases are delivered as reproducible case definitions and documented
  workflows plus tiny deterministic exported fixtures; the heavy solver runs are
  executed by a user in an environment that has the solver.
* Ingestion adapters are implemented with NumPy only (ASCII/CSV/NPZ and a
  minimal legacy-VTK-ASCII reader); binary VTK/HDF5/NetCDF support is optional
  and only added with an explicitly pinned, justified dependency.
* Real public datasets are handled through download scripts with SHA-256
  verification and provenance manifests; large third-party data is not committed
  unless its licence explicitly permits redistribution.

## 1. Common field-data model

A typed, immutable field-data model represents both 2D and 3D velocity fields.

### 1.1 2D field

```
x, y            explicit strictly increasing 1D coordinates (units required)
u, v            velocity components, shape (ny, nx)
pressure        optional scalar, shape (ny, nx)
scalars         optional named scalar fields
time            explicit scalar time (units required) or None for a snapshot
mask            optional boolean validity mask (True = valid)
metadata        immutable provenance/units/convention record
```

### 1.2 3D field

```
x, y, z         explicit strictly increasing 1D coordinates
u, v, w         velocity components, shape (nz, ny, nx)
pressure        optional scalar
scalars         optional named scalar fields
time            explicit scalar time or None
mask            optional boolean validity mask
metadata        immutable provenance/units/convention record
```

### 1.3 Axis and shape convention

2D arrays are `(ny, nx)` (axis 0 = y, axis 1 = x); 3D arrays are `(nz, ny, nx)`
(axis 0 = z, axis 1 = y, axis 2 = x). Coordinates are supplied as explicit 1D
arrays, never inferred. This matches
`itd_research.diagnostics_3d.operators`.

### 1.4 Validation (rejections)

The model rejects: ambiguous axis ordering; mismatched shapes; non-monotonic or
duplicate coordinates where unsupported; NaN/infinity except through an explicit
mask policy; missing units in scientific runs; silently guessed timestep or
boundary conditions. Structured Cartesian grids are supported first; nonuniform
structured grids only where the operators support them correctly. Periodic and
finite domains are both declared explicitly.

The field model is **not** coupled to OpenFOAM, PIV, VTK, or HDF5; those are
handled by adapters that produce the field model.

## 2. Metadata and provenance

Every ingested dataset records: source; author/institution; URL/DOI where
available; licence; version; SHA-256 checksum; date retrieved; original format;
transformations applied; spatial/time/velocity units; coordinate convention;
boundary convention; and uncertainty information when available. A machine
-readable dataset registry (`datasets/registry.json`) holds one entry per
dataset with `redistribution_allowed` set explicitly. Download/preparation
scripts live under `tools/datasets/`, verify checksums, never embed credentials,
never auto-download in CI, and write outside tracked directories by default.

## 3. Established diagnostics (comparison targets)

Implemented (this layer, verified against exact analytical inputs):

* vorticity and vorticity magnitude (2D and 3D);
* strain-rate tensor and its magnitude;
* Q-criterion `Q = 0.5(||Omega||^2 - ||S||^2)` (3D; a 2D analogue for 2x2);
* lambda_2 (Jeong-Hussain), middle eigenvalue of `S^2 + Omega^2`;
* swirling strength (imaginary eigenvalue magnitude of the velocity-gradient
  tensor), 2D and 3D, with an optional swirl-axis and conditioning report;
* Okubo-Weiss (2D);
* enstrophy, palinstrophy, circulation, kinetic energy (existing 2D layer).

For each, the formula, dimensions, sign/threshold convention, derivative method,
boundary treatment, and whether it is local or global are documented in
`docs/research/DIAGNOSTIC_COMPARISON_METHODS.md`. A **local** diagnostic is
compared with a **global** ITD component only through an explicit aggregation
(domain mean, RMS, area/volume fraction above threshold, maximum, percentile,
connected-region count, or integrated positive part).

## 4. Experiments

Reproducible pipelines (real solver/data run outside CI; tiny fixtures in CI):

A. OpenFOAM circular-cylinder wake (Re ~ 40 and ~ 100): vorticity, ITD signature,
   temporal channels, enstrophy/palinstrophy/circulation, shedding frequency and
   Strouhal number (literature comparison kept separate from code verification).
B. Lid-driven cavity (Re 100, 1000, 5000): compare centerline profiles, primary
   and secondary vortices; steady cavities are not treated as temporal-deformation
   validation.
C. Mixing layer / Kelvin-Helmholtz: onset, roll-up, pairing, breakdown phases.
D. Public PIV dataset (selection note first): strict-mask vs documented-repair
   pipelines; interpolated regions never presented as measured data.
E. Vortex transition / merger / splitting: events annotated by criteria
   **independent of ITD** (core count, streamline topology, connected components,
   external labels), then tested against ITD channels.

## 5. Transport versus deformation

For time-resolved external data, at least one transport-compensation method
(exact known displacement for manufactured data; phase correlation; PIV
velocity-based semi-Lagrangian backtracking) separates raw Eulerian temporal
change from transport-aligned change and residual deformation. The residual is
not called a material derivative unless the implementation corresponds to one.

## 6. Experimental 3D ITD candidate

Specified separately in `docs/research/ITD_3D_CANDIDATE_SPEC.md`. The 2D
five-component signature does not generalise by replacing scalar vorticity with
`|omega|`. The candidate studies magnitude-based intensity/heterogeneity/
localization/roughness, an orientation-based replacement for sign mixing, and
complementary helicity and vortex-stretching channels. The outcome may
legitimately be a vector with more than five components. It stays experimental
until validated on genuine 3D CFD or volumetric experimental data.

## 7. Success criteria (hypotheses)

H1 at similar global enstrophy, the ITD structural vector distinguishes fields
with different organization; H2 ITD temporal channels detect annotated
transitions more clearly than intensity alone; H3 transport compensation reduces
false temporal responses from pure translation; H4 ITD components are stable
under reasonable mesh/timestep/PIV-processing changes; H5 ITD is complementary to
Q/swirling/lambda_2 rather than duplicating them; H6 a meaningful 3D extension
requires orientation/stretching/helicity channels. Each is reported as
`supported`, `partially supported`, `not supported`, or `inconclusive`, and it is
stated explicitly which hypotheses can be addressed with the available data
versus which require external CFD/PIV that must be generated or downloaded.

## 8. Decision gates

No new certified revision is created merely because the implementation works. A
revision may be considered only if external CFD is reproducible, at least one
real public PIV dataset is processed, provenance is complete, metrics are stable
under reasonable preprocessing changes, at least one transition is externally
annotated, diagnostic comparisons show complementary information, dimensional
conventions are explicit, all limitations are documented, and independent review
is recommended. The 3D candidate remains experimental until validated on genuine
3D CFD or volumetric experimental data.

## 9. Security and robustness

Archive extraction, malicious paths, decompression bombs, oversized meshes,
malformed VTK/CSV, unsafe pickle/MAT loading, path traversal, symlink overwrite,
non-finite propagation, and eigenvalue failures are all treated defensively.
Python `pickle` is never used for external untrusted data; configurable size
limits and atomic writes are enforced.
